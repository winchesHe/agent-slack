[2026-04-19] M2-renderer Chunk 5 可观测性修复

## 问题
Chunk 5 Observability Checkpoint 中：
  期望：progress message 显示 `🔧 read_file x1`
  实际：progress message 显示 `🔧 bash x1`

## 根因
`src/agent/tools/index.ts` 只注册了 bash / edit_file / save_memory 三个工具，
没有 read_file 工具。模型读文件时走 bash(cat ...)，所以 toolName 是 'bash'。

plan 中的测试用例用 read_file 作为 mock toolName，不反映真实 tool 注册表。

## 处置
用户选择「不新增 read_file 工具」，改为在 progress 中给 bash 工具附带命令信息，
显示为 bash(cmd) x1 格式（如 `bash(cat config.yaml) x1`）。

### 变更文件
- src/agent/AiSdkExecutor.ts
  1. ExecutorStreamPart 的 tool-call 类型增加 args 字段
  2. 新增 toolDisplayLabel() 函数：bash → bash(cmd_truncated)；其他工具保持原名
  3. tool-call-streaming-start 不再发 newToolCalls（此时无 args）
  4. tool-call 始终发 newToolCalls，使用 display label

### 行为变化
- 状态条（assistant.threads.setStatus）仍用原始 toolName: `正在 bash…`
- progress message / finalize 的工具计数使用 display label: `🔧 bash(cat config.yaml) x1`
- newToolCalls 发出时机从 streaming-start 后移到 tool-call（需要 args 才能构建 label）

### 对现有测试的影响
- 测试中无 tool-call-streaming-start 事件，均走 tool-call fallback 路径
- mock toolName 为 read_file（非 bash），toolDisplayLabel 直接返回原名，断言不受影响

[2026-04-19 22:56] 第一轮诊断

## 诊断 log 结论
模型每次 bash 调用都以 `cd /Users/.../agent-slack && <real_cmd>` 开头。
`TOOL_LABEL_CMD_MAX_LEN=40` 截断后，cd 路径吃满额度，真正的命令不可见。
所有调用都显示为 `bash(cd /Users/moego-winches/Desktop/Company/…) xN`。

## 修复
- toolDisplayLabel 新增 CD_PREFIX_RE 正则：`/^cd\s+\S+\s*(?:&&|;)\s*/`
- 先剥掉 cd 前缀，再截断剩余命令
- 预期效果：`bash(cat .agent-slack/config.yaml) x1`

[2026-04-20 00:00] 真实 Slack 测试反馈修复

## 问题 1: invalid_blocks 错误
- `markdownToBlocks` 生成的 blocks 被 Slack API 拒绝
- 加了 fallback 逻辑：blocks 发送失败时降级为纯文本重发

## 问题 2: 终态时 👀 reaction 未移除
- 完成时只加 ✅ 但不移除 👀，导致原消息同时有两个 reaction
- SlackRenderer 新增 `removeAck` 方法（调 `reactions.remove`）
- SlackEventSink finalize 流程：先 `removeAck` 再加终态 reaction（✅/❌/⏹）
- 所有 mock renderer 和集成测试已同步更新

## 问题 3: usage 显示 NaN
- AI SDK `@ai-sdk/openai-compatible` 流式模式不含 `stream_options: { include_usage: true }`
- LiteLLM 不返回流式 usage → provider 默认 NaN → 传播到 formatUsageLine
- 修复 1: `toSafeInt()` 防御性转换，NaN → 0
- 修复 2: streamText 添加 `providerOptions.litellm.stream_options.include_usage = true`
- AiSdkExecutorDeps 新增 `providerName?: string` 字段
- createApplication.ts 传入 `providerName: 'litellm'`

## 问题 4: usage 显示格式优化
- 原格式：`11657 non-cached in+out` → 新格式：`11.7k tokens`
- 新增 `formatTokenCount()` 函数：≥1000 时使用 k 后缀（如 11657 → 11.7k）
- cache 信息仅 cacheHitRate > 0 时才显示
- 已更新 SlackRenderer.test.ts 断言

## 问题 5: providerName 改为环境变量配置
- `createApplication.ts` 从 `process.env.PROVIDER_NAME ?? 'litellm'` 读取
- 同时用于 `createOpenAICompatible({ name })` 和 executor 的 `providerName`
- `.env.example` / `.env` 新增 `# PROVIDER_NAME=litellm` 注释
